# SECURITY REVIEW REPORT — PLAN (Feature 056: Bulk Actions Overhaul)

## Executive Summary

Feature 056 is a **client-side-only** UI refactor that replaces the misleading Publish / Unpublish / Delete bulk-actions dropdown on the Custom Abilities admin page with three ability-native operations (Site Access tri-state, MCP Exposure tri-state, User Access rule apply). Every write path routes through pre-existing, previously-audited REST endpoints:

- `POST /acrossai-abilities-manager/v1/abilities/{slug}` — owned by `AcrossAI_Abilities_Write_Controller`; enforces `manage_options` + `X-WP-Nonce`; sanitises via `AcrossAI_Sanitizer::sanitize_tri_state`.
- `PUT /wpb-ac/v1/abilities/rules/acrossai-abilities/{slug}` — owned by the `wpboilerplate/wpb-access-control` composer package; capability + nonce enforcement is upstream contract.

**No new authorisation surface, no new database tables, no new REST endpoints, no new PHP code, no new composer or npm dependencies.** Server-side sanitisation, escaping, capability checks, and slug validation (`SEC-01`) are unchanged. Per project policy (see `~/.claude/.../feedback_skip_permission_callback_audit.md`) `permission_callback` compliance on the preserved routes is intentionally not re-audited in this review — the feature adds no new route to gate.

**Overall risk: LOW.** Two Low-severity findings pertain to client-side failure-visibility hygiene (unhandled promise rejection on partial failures + missing null-response check on the composer PUT). Three informational findings note deferred concerns (unbounded concurrency, defensive URL encoding parity, and Jest test coverage completeness). No Critical, High, or Medium issues surfaced.

**Recommendation**: proceed to `/speckit-tasks` with the two Low findings absorbed into TASK-3 (`handleBulkApply`), TASK-5 (`UserAccessBulkModal.handleApply`), and TASK-7 (Jest coverage). No blocking issues; no follow-up remediation feature required.

## Vulnerability Findings

### SEC-001 — Partial-failure silent success in `handleBulkApply()` tri-state paths

- **Severity**: LOW
- **OWASP Category**: A04:2025 — Insecure Design
- **CWE**: CWE-703 — Improper Check or Handling of Exceptional Conditions
- **CVSS v3.1**: 3.1 (AV:N / AC:H / PR:H / UI:R / S:U / C:N / I:L / A:N)
- **Location** (planned): `src/js/abilities/components/AbilitiesList.jsx` — the two `await dispatch(bulkUpdateTristate(...))` calls inside `handleBulkApply()`
- **Spec-Kit Task**: TASK-SEC-001 → absorb into feature TASK-3

**Observation**: The `handleBulkApply` snippet in `docs/planning/056-bulk-actions-overhaul.md` (TASK-2) awaits the tri-state dispatch and then clears selection unconditionally:

```js
if (domain === 'site_access') {
  const v = value === 'force_allow' ? true : value === 'force_block' ? false : null;
  await dispatch(bulkUpdateTristate(selectedSlugs, 'site_allowed', v));
} else if (domain === 'mcp') { ... }

setBulkAction('');
setSelectedSlugs([]);
```

`bulkUpdateTristate` wraps `Promise.all(slugs.map(...))` — if any per-slug write rejects, the outer promise rejects, the `await` throws, the two `setState` calls do not execute, and the operator sees no error banner because there is no `try/catch` at this call site. The admin's selection stays intact by accident, but the mistake is that failures are not surfaced. The operator may believe the bulk apply succeeded on slugs 1–5 when only 3 completed and 2 rejected (e.g. a nonce race, a per-slug validation failure, or a temporary REST failure).

Data-integrity impact is bounded — server state is consistent per slug (each per-slug write is atomic on the server); only the *client's* perception of success is unreliable. Because access to this page requires `manage_options`, the attacker surface is nil; risk is operator confusion, not exploitation.

**Remediation** (fold into TASK-3):
- Wrap the two tri-state dispatches in `try / catch`.
- On catch, keep the selection intact, keep the dropdown value, and surface the error via the existing store-level error path (or a `wp.data.dispatch('core/notices').createErrorNotice(...)`).
- Same discipline the User Access modal already applies at `handleApply` (`try/catch` + `setError(...)`).

### SEC-002 — Composer PUT null-response not treated as failure

- **Severity**: LOW
- **OWASP Category**: A04:2025 — Insecure Design
- **CWE**: CWE-754 — Improper Check for Unusual or Exceptional Conditions
- **CVSS v3.1**: 2.5 (AV:N / AC:H / PR:H / UI:R / S:U / C:N / I:L / A:N)
- **Location** (planned): `src/js/abilities/store/index.js` — new `bulkSetUserAccessRule` thunk; also `UserAccessBulkModal.jsx` — `handleApply`
- **Spec-Kit Task**: TASK-SEC-002 → absorb into feature TASK-4 and TASK-5

**Observation**: The composer's `PUT /wpb-ac/v1/abilities/rules/acrossai-abilities/{slug}` endpoint has been observed to return `null` on soft failures rather than throwing (root of `BUG-AC-NULL-RETURN-SILENT-FAIL`, still marked Active). The planned thunk does not inspect per-slug responses:

```js
bulkSetUserAccessRule(slugs, acKey, acOptions) {
  return async ({ dispatch }) => {
    await Promise.all(slugs.map((slug) => apiFetch({ ... })));  // ← no null-check on the results
    dispatch(actions.fetchAbilities());
  };
}
```

If some slugs silently return `null`, `Promise.all` resolves (no rejection was thrown), `dispatch(actions.fetchAbilities())` runs, and the modal's `handleApply` proceeds to `onApplied()` — closing the modal and clearing selection. The operator sees "applied" while some rule rows were not written.

Data-integrity impact identical to SEC-001: server state is per-slug consistent (a null-return means "not written" on that slug); only client perception is misleading. Admin-only surface.

**Remediation** (fold into TASK-4 + TASK-5):
- In `bulkSetUserAccessRule`, capture `Promise.all` results and assert each is non-null (or has the expected shape returned by the composer's `PUT`). Reject with a synthetic Error naming the failed-slug count if any are null.
- In `UserAccessBulkModal.handleApply`, the existing `try/catch` will surface this rejection via `setError(...)` — no change needed there once the thunk propagates.
- Add a Jest test in `store.bulkSetUserAccessRule.test.js` (TASK-7) that mocks one slug returning `null` and asserts the thunk rejects.

### SEC-003 — Unbounded `Promise.all` concurrency on bulk apply

- **Severity**: INFORMATIONAL
- **OWASP Category**: A08:2025 — Software and Data Integrity Failures
- **CWE**: CWE-770 — Allocation of Resources Without Limits or Throttling
- **CVSS v3.1**: 0.0 (documented deliberate behaviour)
- **Location** (planned): `src/js/abilities/store/index.js` — `bulkUpdateTristate` and `bulkSetUserAccessRule`
- **Spec-Kit Task**: TASK-SEC-003 → **no code change** — recorded for future revisit

**Observation**: Both new thunks fire N parallel writes with a single `Promise.all` — no chunking, no concurrency cap. This is deliberate: the plan's Technical Context and the user brief both mandate mirroring the existing `bulkUpdateStatus` shape at `src/js/abilities/store/index.js:284-299`. The rationale is consistency-preserving and admin-scoped (`manage_options` gate); the surface is limited to trusted operators.

Potential concern for future revisit only: on a selection of 200+ abilities, N concurrent same-origin writes may transiently degrade admin responsiveness (browser connection cap + PHP-FPM slot pressure). The existing `bulkUpdateStatus` has never been exercised at that scale in production so no telemetry justifies a change now. If future usage grows, a p-limit-style chunker (5-10 in flight) would be a drop-in.

**Remediation**: None for this feature — documented as accepted concurrency profile. If the future acrossai-mcp-manager or acrossai-abilities-log companion plugin surfaces a "bulk apply on 1000 slugs" workflow, revisit both this thunk pair *and* the reference `bulkUpdateStatus` together (one utility, three consumers).

### SEC-004 — Client-side slug URL encoding parity for `api.updateAbility`

- **Severity**: INFORMATIONAL
- **OWASP Category**: A04:2025 — Insecure Design (defence-in-depth)
- **CWE**: CWE-20 — Improper Input Validation
- **CVSS v3.1**: 0.0 (server sanitises regardless; SEC-01 covers this)
- **Location** (planned): `src/js/abilities/store/index.js` — `bulkUpdateTristate` (delegates to `api.updateAbility(slug, ...)`)
- **Spec-Kit Task**: TASK-SEC-004 → verify during feature TASK-4

**Observation**: The new `bulkSetUserAccessRule` thunk explicitly `encodeURIComponent(slug)` before interpolating into the REST path — good. The new `bulkUpdateTristate` thunk delegates to the existing `api.updateAbility(slug, ...)` helper, whose slug-encoding discipline this review has not verified end-to-end from planning artifacts alone. Given that server-side `sanitize_ability_slug()` (`SEC-01`) already normalises the slug regardless, this is defence-in-depth only.

**Remediation** (fold into TASK-4): read `src/js/abilities/store/index.js` `api.updateAbility` definition; if it does not `encodeURIComponent(slug)` before URL construction, add it there (single change, benefits every existing consumer). No new code path required if the helper already handles it.

### SEC-005 — Jest coverage for security-critical branches

- **Severity**: INFORMATIONAL
- **OWASP Category**: A09:2025 — Security Logging and Monitoring Failures
- **CWE**: CWE-1053 — Missing Documentation of Design (test coverage as documentation)
- **CVSS v3.1**: 0.0
- **Location** (planned): `tests/jest/abilities/` — the three new test files planned in TASK-7
- **Spec-Kit Task**: TASK-SEC-005 → confirm during feature TASK-7

**Observation**: TASK-7 plans three new Jest suites. This review confirms the *scope* of those suites should explicitly include:

- `store.bulkUpdateTristate.test.js` — cover: (a) success path with N slugs; (b) partial-failure path where 1-of-N rejects (asserts SEC-001 remediation surfaces error); (c) tri-state JSON payload discipline (assert body sends raw `true`/`false`/`null`, not string aliases — guards `BUG-MERGER-BOOL-STRING-CAST`).
- `store.bulkSetUserAccessRule.test.js` — cover: (a) success path; (b) null-response path (asserts SEC-002 remediation rejects); (c) URL encoding (assert one slug with `/` is properly encoded).
- `UserAccessBulkModal.test.jsx` — cover: (a) provider select renders the dynamic list from the plugin adapter; (b) Apply on the "Everyone (clear rule)" sentinel dispatches with empty `ac_key` + empty `ac_options`; (c) error state renders when thunk rejects.

Test-environment plumbing constraints (per plan §T007 guards): `PATTERN-JESTENV-WPSCRIPTS`, `BUG-WP-ELEMENT-ACT-MISSING`, `BUG-MODULE-LEVEL-WINDOW-READ`, `BUG-JEST-ASYNC-USEEFFECT-FLUSH`, `BUG-WP-API-FETCH-VIRTUAL`.

**Remediation**: none — informational. The TASK-7 owner should treat this bullet list as the acceptance criteria for that task.
