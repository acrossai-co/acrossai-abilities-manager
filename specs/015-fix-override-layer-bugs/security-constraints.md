## Trust Boundaries

All PHP changes operate within existing trust boundaries:
- **REST `permission_callback`** remains the sole authorization gate for all endpoints (unchanged). `manage_options` is enforced before `save_override()` is called. No new endpoints or capability checks are introduced.
- **DB-layer helpers** (`get_ability_by_id`, `save_override`, etc.) are authorization-free per DEC-BY-SOURCE-AUTHZ; the REST controller is the gate. T004/T005 do not change this contract.

## Authorization Assumptions

- T005 removes the `get_override_by_slug()` re-query from the Write Controller non-db path. This is a positive security improvement: the stale-cache null return could have led to incorrectly representing `has_override: false` to the client.
- Error code change (`rest_update_failed` → `save_override_failed`) is information-neutral — the HTTP status remains 500.

## Watchpoints for Implementation

1. **T004 (INSERT path)**: Null-assignment block (`callback_type = null`, `status = null`) MUST be placed **before** `$this->add_item($fields)`, not after.
2. **T004 (SEC-002)**: The `if ( array_key_exists( 'source', $fields ) && 'db' === $fields['source'] )` guard must remain at the **top** of `save_override()`, before `prepare_fields_for_write()`.
3. **T005**: After the `$override_row` variable rename (from `$saved`), verify the `if ( false === $override_row )` check uses strict comparison (`===`), consistent with SEC-04 (strict type comparison pattern).
4. **T001**: `is_array()` guard on `$mcp_meta` must use PHP `is_array()`, not `isset()` alone, to handle the case where `get_meta_item('mcp')` returns a non-array value.
