# Implementation Plan: Abilities Library Admin Submenu

**Branch**: `027-keys-submenu` | **Date**: 2026-06-06 | **Spec**: `specs/027-keys-submenu/spec.md`
**Input**: Feature specification from `/specs/027-keys-submenu/spec.md`

## Summary

Add a "Library" admin submenu page under the Abilities Manager top-level menu. The page renders a DataViews grid of `main_key` ability-group cards, each with an ON/OFF master toggle, an All/Specific mode selector, and (in Specific mode) per-`sub_key` checkboxes. Add-ons register ability definitions via an `acrossai_abilities_api_init` filter hook; the plugin collects those definitions at `init`, applies saved toggle config, and registers only approved abilities at `wp_abilities_api_init`. Toggle state is stored as a site option (sparse: only non-default entries) and exposed via a new Library REST controller under the dedicated `acrossai-abilities-library/v1` namespace. The existing Logger logs namespace must move from `acrossai-abilities/v1` to `acrossai-abilities-log/v1`.

## Key Design Decisions

### D1: Dedicated Library REST Namespace
The new config endpoint uses `acrossai-abilities-library/v1`, not the existing manager namespace and not the logger namespace. The endpoint is `/acrossai-abilities-library/v1/abilities/config`.

### D2: Logger Namespace Migration
`includes/Modules/Logger/Rest/AcrossAI_Logger_Logs_Controller.php` currently uses `protected $namespace = 'acrossai-abilities/v1';`. This must change to `protected $namespace = 'acrossai-abilities-log/v1';` so logger logs have a dedicated namespace and do not collide with the Library module.

### D3: Registry Fires Filter at init P99
`AcrossAI_Ability_Library_Registry` fires the `acrossai_abilities_api_init` filter at `init` priority 99. Add-ons hook at standard `init` priority (10), giving them time to register before collection. Results are cached for the request, making all subsequent calls idempotent. The Processor consumes the cached collection at `wp_abilities_api_init`.

### D4: Definitions Localized to React, Not REST-Exposed
`LibraryMenu::render()` localizes the collected definitions (from `AcrossAI_Ability_Library_Registry::get_definitions()`) into `window.acrossaiAbilityLibraryData` via `wp_localize_script`. Definitions only change when add-ons activate/deactivate (which causes a page reload anyway). This avoids a second REST roundtrip on page load and keeps the config endpoint focused on state only.

### D5: Config Site Option Structure
```json
{
  "content": {
    "enabled": false,
    "mode": "all",
    "sub_keys": {}
  },
  "media": {
    "enabled": true,
    "mode": "specific",
    "sub_keys": {
      "summarize": true,
      "generate": false
    }
  }
}
```
Keys are sanitized with `sanitize_key()`. `mode` is validated against `['all', 'specific']`. `enabled`/`sub_key` values are cast to strict booleans. Only non-default entries are written (see D6).

### D6: Default Config Behavior — Sparse Storage (FR-017, FR-018)
A `main_key` absent from saved config defaults to `enabled=true, mode='all'`. A `sub_key` absent from saved config in Specific mode defaults to `enabled=false`. Applied in the Processor's `is_permitted()`, not the Config utility.

**Sparse storage rule**: `save_config()` strips any entry where `enabled=true AND mode='all' AND sub_keys=[]` before writing to the site option. This keeps `acrossai_library_config` lean — only entries that deviate from the default are persisted. The Processor's absent-key path is the canonical source of truth for default behavior.

```php
// Inside save_config() after sanitization, before update_site_option():
foreach ( $sanitized as $key => $entry ) {
    if ( true === $entry['enabled'] && 'all' === $entry['mode'] && empty( $entry['sub_keys'] ) ) {
        unset( $sanitized[ $key ] );
    }
}
```

### D7: is_library_page() Hook Suffix Formula
```php
private function is_library_page( string $hook_suffix ): bool {
    return 'acrossai-abilities-manager_page_acrossai-abilities-library' === $hook_suffix;
}
```
Formula: `{parent-slug}_page_{submenu-slug}`. Submenu slug: `acrossai-abilities-library`. (DEC-MENU-HOOK-SUFFIX)

### D8: LibraryMenu Uses Singleton Pattern (No get_hook_suffix())
`LibraryMenu` does NOT store the return value of `add_submenu_page()`. The `is_library_page()` guard in `admin/Main.php` uses the hardcoded literal. This avoids the existing `LogsMenu::get_hook_suffix()` anti-pattern.

### D9: Abstract `Ability_Definition` Class as Add-on Authoring API

`includes/Modules/Library/Ability_Definition.php` provides a class-based alternative to the raw `acrossai_abilities_api_init` filter. Add-on ability authors extend it and implement five abstract methods instead of writing boilerplate filter callbacks.

```php
abstract class Ability_Definition {
    public function __construct() {
        add_filter( 'acrossai_abilities_api_init', array( $this, 'push_definition' ) );
    }
    abstract protected function main_key(): string;
    abstract protected function main_key_label(): string;
    abstract protected function sub_key(): string;
    abstract protected function sub_key_label(): string;
    abstract protected function ability(): array; // returns ['name' => ..., 'args' => [...]]

    public function push_definition( array $definitions ): array { ... }
}
```

**Why a class and not just the filter**: each class encapsulates its own `execute()` method, making `execute_callback => [$this, 'execute']` possible. No global function naming required.

**Instantiation pattern**: add-ons call `new MyAbility()` inside `plugins_loaded` priority 20, guarded by `class_exists('\\AcrossAI_Abilities_Manager\\Includes\\Modules\\Library\\Ability_Definition')`. This ensures no fatal if the manager is inactive and fires well before the Registry collects at `init P99`.

**Raw filter remains valid**: add-ons that prefer procedural PHP or do not use Composer autoloading can still push definitions directly onto `acrossai_abilities_api_init`. Both paths produce identical definition arrays consumed by the Registry.

**Companion add-on migration** (`acrossai-core-abilities`): the three procedural ability files (`transient-flush.php`, `debug-log-reader.php`, `debug-log-reset.php`) were converted to `includes/Abilities/Transient_Flush.php`, `Debug_Log_Reader.php`, and `Debug_Log_Reset.php` extending `Ability_Definition`. `Main::register_abilities()` bootstraps them at `plugins_loaded P20`.

### 1.4 — PHP: Library REST Orchestrator + Config Sub-controller

**SC-027-06 (REQUIRED)**: `AcrossAI_Ability_Library_Rest_Controller` MUST be a **standalone singleton** — it does NOT extend `WP_REST_Controller`. Follow the `AcrossAI_Abilities_Rest_Controller` pattern exactly, not the `AcrossAI_Logger_Controller` pattern (which extends `WP_REST_Controller` and inherits a different permission model).

Canonical orchestrator structure:
```php
class AcrossAI_Ability_Library_Rest_Controller {
    const REST_NAMESPACE = 'acrossai-abilities-library/v1';
    protected static $instance = null;
    public static function instance(): self { ... }
    private function __construct() {}

    public function register_routes(): void {
        AcrossAI_Ability_Library_Config_Controller::instance()->register_routes();
    }

    public function check_permission( \WP_REST_Request $request ) {
        if ( ! current_user_can( 'manage_options' ) ) {
            return new \WP_Error( 'rest_forbidden',
                __( 'You do not have permission to manage abilities.', 'acrossai-abilities-manager' ),
                array( 'status' => 403 ) );
        }
        $nonce = $request->get_header( 'X-WP-Nonce' );
        if ( ! $nonce || ! wp_verify_nonce( $nonce, 'wp_rest' ) ) {
            return new \WP_Error( 'rest_forbidden',
                __( 'Nonce verification failed.', 'acrossai-abilities-manager' ),
                array( 'status' => 403 ) );
        }
        return true;
    }
}
```

`AcrossAI_Ability_Library_Config_Controller` (singleton sub-controller):

```
GET  /acrossai-abilities-library/v1/abilities/config  → get_config()
POST /acrossai-abilities-library/v1/abilities/config  → save_config()
Both: permission_callback = array( AcrossAI_Ability_Library_Rest_Controller::instance(), 'check_permission' )
```

GET returns current config (reads site option).
POST validates + sanitizes request body, saves via `AcrossAI_Ability_Library_Config::save_config()` (strips defaults before write), returns saved config.

## Complexity Tracking

| Decision | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `acrossai_abilities_api_init` filter (public hook) | Loose coupling: add-ons register without plugin-class dependency (FR-010) | Direct PHP API call would create tight coupling |
| Registry fires at `init P99` | Definitions must be available for gating AND admin localization while giving add-ons maximum `init` flexibility | Firing at `wp_abilities_api_init` would miss admin-page localization window |
| Sparse config storage (D6) | `acrossai_library_config` should only grow when admin deviates from defaults; absent key = enabled is the canonical default (FR-017) | Storing all entries (including defaults) bloats the option and obscures intent |
| Config REST sub-controller registered first in orchestrator | Prevents wildcard shadowing (BUG-REST-ROUTE-ORDER-LITERAL-BEFORE-WILDCARD) | Not registering first would cause `/abilities/config` to be shadowed by slug route |
